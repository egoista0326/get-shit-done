## Confirmed Decisions

- New milestone version: **v2.0**.
- Phase numbering: **reset to Phase 01** for the rebuild.
- Current v1.4 disposition: **`pivoted_not_shipped`**; do not spend more effort completing the current ljx-GSD review loop.

- Auto/ARIS capabilities should be exposed as standalone `gsd` commands, but internally they should attach through special research-phase execution conventions with minimal interference to normal GSD.
- Do **not** add `phase_type`, typed phase routing, or any broad phase schema extension. The previous typed-phase approach had too much blast radius.

## Milestone Intent

Rebuild the current `ljx-GSD` direction into a minimal-modification, framework-first `gsd` implementation that keeps upstream GSD as the outer lifecycle/control-plane base and integrates Auto/ARIS research as a bounded native workflow extension.

The milestone should not start by coding. It must first extract durable frameworks for upstream GSD, current ljx-GSD/history, and Auto/ARIS; then design and adversarially review the target GSD framework; only then create a clean repo copy and begin incremental implementation by copying/reusing upstream GSD and applying small, reviewed modifications.

## Non-Negotiable Principles

- **GSD-first:** The outer lifecycle, milestone/phase semantics, state discipline, planning artifacts, git behavior, review gates, workstreams, and hooks should remain GSD-shaped unless there is a documented reason to deviate.
- **Minimal modification:** Prefer copying, wrapping, or extending upstream GSD behavior over rewriting it. Every divergence needs rationale and a regression expectation.
- **Auto as command surface plus special phase convention:** Auto/ARIS research should be exposed as standalone `gsd` commands, but internally those commands operate through ordinary GSD phases plus a narrow research-phase convention. They must not create a second control plane, second state root, or separate completion semantics.
- **No phase type expansion:** Do not add `phase_type`, typed phase routing, or global phase schema changes. A research phase is identified by its command/artifact contract and plan content, not by a new authoritative phase-type field.
- **Framework before implementation:** Implementation starts only after the target framework and review rubric have been discussed and accepted.
- **Evidence before completion:** A phase or research step cannot be marked complete unless the required execute evidence exists.
- **Serial authoritative writes:** Lifecycle/state synchronization should be serialized or lock-protected. No parallel agents may write the same authoritative state family concurrently.
- **Self-contained but capability-preserving:** The final `gsd` package should not depend on globally installed upstream GSD/Auto skills at runtime, but it must preserve their substantive capabilities through bundled/internal assets and prompts.
- **Git parity:** Research flows must follow GSD-like git, summary, review, and verification behavior where they mutate project artifacts.
- **Clean implementation copy:** Implementation must occur in a clean repo copy/worktree after framework approval. The current repository remains the planning and historical evidence workspace until that point.
