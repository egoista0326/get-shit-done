# 01-GSD-UPGRADE-BOUNDARIES

**Status:** build-round draft
**Evidence boundary:** Synthesized from upstream GSD subagent reports.

## Compatibility Boundaries

| Boundary | Why it matters | v2.0 rule |
|---|---|---|
| `.planning/` artifact names | Workflows and runtime helpers consume file names, frontmatter, and artifact presence. | Preserve unless a reviewed migration bridge exists. |
| `PROJECT.md`, `REQUIREMENTS.md`, `ROADMAP.md`, `STATE.md` | These are active runtime inputs and human-readable mirrors. | Treat them as runtime APIs. Do not let multiple writers update them concurrently. |
| Phase artifact names | Plan/summary/context/verification naming drives roadmap and phase analysis. | Maintain naming compatibility or build a proven adapter. |
| `gsd-tools.cjs` CLI | Workflows, SDK, and scripts call it as a deterministic adapter. | Preserve command names, `--cwd`, `--ws`, `--raw`, `--pick`, `--default`, JSON shapes, and `@file` spillover unless intentionally versioned. |
| Workstream routing | Workstream scope changes all planning paths. | Preserve explicit `--ws`, environment variable, session pointer, shared pointer, and flat fallback semantics. |
| Planning locks and state locks | Parallel work depends on safe write serialization. | Preserve equivalent planning lock, state lock, and atomic writes. |
| Orchestrator/subagent ownership | Avoids worktree and shared artifact races. | Orchestrator owns lifecycle state; executor agents own scoped implementation and summary artifacts; all canonical lifecycle writes go through one owner and lock path. |
| Git planning commits | Planning commits, branch strategies, and subrepo routing are user-visible. | Preserve `commit_docs`, gitignore checks, phase/milestone branch templates, subrepo commit routing, and explicit file staging semantics. |
| Hook ownership and uninstall symmetry | Runtime install/uninstall must not corrupt user config. | Preserve marker-managed config sections, hook ownership tracking, and user artifact preservation. |
| Agent names and profiles | Workflows reference concrete agent roles. | Preserve names or provide reviewed aliases. |
| Long workflow prompts | Copying prompt bodies creates upgrade drift. | Index prompt bodies by path and extract only stable contracts. |
| Package manifests and test corpus | `package.json`, `sdk/package.json`, build scripts, and regression tests define publish/install/build behavior. | Treat manifests and package/install/hook/atomic-write tests as compatibility inputs before changing packaging. |
| SDK package boundary | The SDK has its own public API, CLI entrypoint, prompt assembly, and tests. | Decide whether v2.0 consumes, adapts, or defers the SDK explicitly; do not assume it is only a wrapper. |

## Upgrade Risks Extracted By Subagents

| Risk | Source-extracted evidence | Phase 02 implication |
|---|---|---|
| Reference vs installed version divergence | Reference checkout is `1.35.0`; installed local runtime reports `1.34.2`; sampled installed workflow files differ. | Decide the canonical baseline before implementation. |
| Config reads can mutate files | `loadConfig()` writes migrations and subrepo syncs. | Decide if v2.0 keeps hidden migration-on-read or requires explicit migration. |
| Config validation drift | Template/docs include keys not in `VALID_CONFIG_KEYS`; hooks read keys not validated. | Require config model review before copying config code. |
| Model profile drift | Docs mention `inherit`; runtime profiles include `quality`, `balanced`, `budget`, `adaptive`; SDK includes `speed`. | Normalize model profiles and reviewer policy before implementation. |
| Installer monolith | Upstream `bin/install.js` owns many unrelated runtime adapters. | Consider decomposing adapters before adding research behavior. |
| Hook copy/registration ambiguity | Codex update hook may register a file path not copied by generic hook logic, based on subagent inference. | Review install/hook behavior in implementation planning. |
| Markdown state brittleness | `STATE.md` frontmatter is rebuilt from regex/markdown and disk. | Avoid expanding state semantics without schema review. |
| Agent inventory mismatch | There are more `gsd-*.md` agent files than profiled agents. | Determine authoritative agent inventory. |
| Missing explicit git tag behavior | Subagent found branch/commit behavior, not tag management. | Do not assume milestone tags are upstream behavior. |
| Generated hooks absent in source tree | `hooks/dist` is generated by build scripts. | Packaging must run build/prepublish checks before install verification. |
| Package/test surface underindexing | R1 found package manifests and key install/hook/atomic-write tests were missing from the initial source index. | Keep package and regression corpus in the framework review matrix before implementation. |
| SDK scope ambiguity | R1 found SDK public API, CLI, prompt templates, and tests were too narrowly described. | Resolve SDK inclusion before Phase 06 foundation work. |

## Recommended v2.0 Baseline Policy

1. Treat upstream GSD reference checkout as the source baseline unless Phase 02 chooses installed `1.34.2` or a reconciled diff.
2. Keep v2.0 changes as adapters around upstream GSD behavior, not rewrites of lifecycle semantics.
3. Preserve `.planning/` compatibility until a reviewed alternative proves better.
4. Keep research integration outside core phase schema; expose it as standalone commands plus artifact conventions.
5. Do not add research-specific state families to core GSD unless review proves they are necessary and bounded.
6. Do not copy prompt bodies wholesale; keep prompt source indexes stable for future upstream upgrade.
7. Require review of config, hooks, installer, and state ownership before code implementation.
8. Require a single-writer map for canonical lifecycle families before implementation. Derived mirrors and imported research copies must be read-only until adopted through the lifecycle owner.
