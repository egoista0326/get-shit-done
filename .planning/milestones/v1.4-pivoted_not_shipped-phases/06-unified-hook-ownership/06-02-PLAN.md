# Phase 6 Plan 06-02: Artifact And Integration Hook Contract

**Date:** 2026-04-09
**Status:** Draft execution-grade plan

## Objective

Turn the accepted Phase 6 render-hook and integration-hook boundaries into an implementation-grade contract that later `ljx-GSD-*` rewrite work can follow without reopening ownership questions or inventing a second control plane.

## Locked Assumptions

- `.planning/` remains the authoritative root.
- `06-01` lifecycle ownership remains locked: commands/workflows own lifecycle truth.
- `05-02` state-family namespaces and report contracts remain locked; this plan must not invent a new parallel state family unless a later accepted decision explicitly reopens schema.
- GSD's existing implementation should remain the preferred reference wherever it already solves installer/runtime/helper separation cleanly.
- `planning-with-files` remains useful as an observer/reminder-hook reference only, not as an ownership model.
- Auto logging and integration patterns remain useful as evidence emitters only, not as lifecycle owners.

## Scope Boundary

This plan is responsible for:

- defining which render hooks may create or refresh derived artifacts
- defining which narrow metadata fields render hooks may mirror back
- defining where integration facts and local integration-object status should land
- defining which render/integration actions should remain command-invoked runtime helpers instead of always-on hooks
- defining failure, freshness, idempotency, and logging expectations for render and integration hooks

This plan is not responsible for:

- reopening lifecycle ownership from `06-01`
- reopening canonical state-family namespaces from `05-02`
- defining Codex adapter/runtime-helper ownership in full detail (reserved for `06-03`)
- implementing hook scripts yet

## Reference Implementation Bias

When choices are available, prefer GSD-style implementation patterns:

1. keep the global hook surface small
2. prefer explicit command/workflow invocation of runtime helpers over ambient always-on hooks
3. keep hook registration explicit and idempotent
4. keep derived artifact generation separate from control-truth mutation
5. keep external integrations evidence-oriented rather than workflow-owning

Concrete reference signals:

- GSD's visible Codex installation keeps hook usage light and explicit
- GSD's installer already separates command/workflow meaning from runtime translation and hook registration
- `planning-with-files` demonstrates useful reminder-hook patterns, but is intentionally more aggressive than the ownership model desired here
- Auto shows valuable evidence/log/state-emitter patterns, but its loop-state files must not be reintroduced as a parallel control plane

## Execution Order

1. **WP1: Render hook allow/deny contract**
2. **WP2: Integration evidence and local-status contract**
3. **WP3: Registration, invocation, and failure-handling contract**
4. **Integration pass: check alignment with `06-01`, `05-02`, and the GSD-reference implementation bias**

Reasoning:

- WP1 fixes what render hooks may produce and mirror.
- WP2 fixes what integrations may report and where those reports land.
- WP3 prevents implementation from overusing ambient hooks when a command-invoked helper is the safer GSD-style shape.
- The integration pass verifies that none of this drifts back into state-steering behavior.

## Work Package 1: Render Hook Allow/Deny Contract

**Goal**

Define a stable contract for render-style automation such as summary refresh, review artifact refresh, compile/render outputs, export views, and migration/operator reports.

**Concrete steps**

1. Convert the accepted render-hook matrix into an implementation-facing allow/deny table.
2. Lock the canonical render families for the first rewrite wave:
   - summary/report refresh
   - code-review artifact refresh
   - paper compile/render outputs
   - migration/operator report refresh
   - visual/export report refresh
3. Define exactly which metadata fields count as narrow mirror-back:
   - `render_status`
   - `rendered_at`
   - `artifact_path`
   - `artifact_hash`
   - trace/log references
4. Define freshness and idempotency expectations:
   - repeated render calls should be safe
   - a refreshed render may supersede prior render metadata
   - render outputs must never silently overwrite control truth
5. Define the placement rule:
   - render artifacts live in their canonical artifact/report locations
   - mirror-back metadata lands only in existing structured families already locked by `05-02`
   - do not create a new top-level render state family in this phase

**Expected outputs**

- render-hook allow/deny contract
- narrow metadata mirror-back definition
- render freshness and idempotency contract
- list of forbidden render-side writes

**Validation**

- no render hook is allowed to mutate active phase, current position, roadmap, workstream pointer, or any domain verdict
- every allowed render metadata field is clearly non-semantic
- every render output path maps cleanly onto already-accepted artifact/report locations

## Work Package 2: Integration Evidence And Local-Status Contract

**Goal**

Define how experiment, remote execution, compile, notification, and external-tool integrations feed facts back into `ljx-GSD` without becoming lifecycle owners.

**Concrete steps**

1. Convert the accepted integration matrix into explicit evidence/object-status/forbidden-write tables.
2. Lock the first-wave integration families:
   - experiment monitoring and run-tracking
   - remote execution / deploy
   - paper compile
   - notification integrations
   - external benchmark or report tools
3. Define where each class of integration output lands:
   - experiment facts attach to `.planning/state/experiments/{phase_id}.json` and phase-local experiment artifacts
   - review/claim follow-up facts attach only through command/workflow interpretation, not direct hook verdict writes
   - paper compile facts attach to `.planning/state/papers/{phase_id}.json` and paper-local outputs
   - notification and remote-execution traces attach as evidence/trace metadata, not lifecycle truth
4. Define the allowed local-status boundary:
   - integration hooks may write object-local statuses such as `running`, `failed`, `completed`, `sent`, `compile_failed`
   - they may not escalate those statuses into `phase complete`, `claim supported`, `verify passed`, or `next action`
5. Define the interpretation handoff:
   - commands/workflows remain the only layer allowed to interpret integration evidence into project decisions

**Expected outputs**

- integration evidence contract
- local integration-object status contract
- evidence-to-workflow interpretation handoff rule
- forbidden verdict-write table

**Validation**

- no integration output path becomes a second lifecycle owner
- all allowed writes remain factual, local, and evidence-oriented
- all domain verdicts remain workflow-owned

## Work Package 3: Registration, Invocation, And Failure-Handling Contract

**Goal**

Decide which artifact/integration behaviors should be ambient hooks at all, and which should instead stay as explicit runtime-helper calls initiated by commands/workflows.

**Concrete steps**

1. Define the default GSD-style preference:
   - command-invoked helper first
   - ambient hook only when continuous observation, warning, or lightweight trace emission is actually needed
2. Split the first rewrite wave into:
   - always-safe observer/guard hooks
   - explicit render helpers
   - explicit integration sync helpers
3. Define failure handling:
   - hook failure may mark render/integration status as failed
   - hook failure may emit evidence and warnings
   - hook failure may not invent a new lifecycle verdict
4. Define logging and trace expectations:
   - render/integration runs should leave enough trace metadata for later diagnosis
   - traces remain evidence, not control truth
5. Define escalation expectations:
   - unresolved render/integration failures may block only through already-accepted gates
   - otherwise they warn and attach evidence for command/workflow interpretation

**Expected outputs**

- render/integration registration strategy
- explicit helper-vs-hook split for the first rewrite wave
- failure-handling contract
- trace/logging contract

**Validation**

- the resulting design stays closer to GSD's light-hook model than to `planning-with-files`'s aggressive reminder-hook model
- ambient hooks are used only where they provide clear runtime value
- helper/hook separation remains compatible with later `06-03` adapter work
