# Phase 6 Plan 06-01: Hook Ownership Contract And Implementation Handoff

**Date:** 2026-04-09
**Status:** Draft execution-grade plan

## Objective

Turn the accepted Phase 6 hook-ownership semantics into an implementation-grade contract so later `ljx-GSD-*` rewrite work can proceed without reopening ownership boundaries.

## Locked Assumptions

- `.planning/` remains the authoritative root.
- Phase 5 migration and cutover semantics remain locked and must not be reopened here.
- `ljx-GSD-code-review` remains a distinct quality gate.
- GSD's existing implementation should be the preferred reference whenever it already solves installer/runtime separation cleanly.

## Work Package 1: Lifecycle Ownership And Guard Contract

**Goal**

Specify, in implementation-facing form, that lifecycle truth belongs to commands/workflows while hooks only guard or annotate.

**Reference implementation bias**

- prefer GSD's current command/workflow ownership model
- use hooks only where GSD already treats them as runtime guards or utilities, not as mainline progression owners

**Concrete steps**

1. Convert the accepted lifecycle ownership matrix into a stable implementation table.
2. Define which lifecycle actions may be blocked by hooks and under what rule source.
3. Define which lifecycle actions may never be hook-owned.
4. Define the minimal structured state fields later commands should read instead of re-parsing prose artifacts.

**Expected outputs**

- lifecycle ownership contract
- lifecycle guard contract
- list of forbidden hook-owned lifecycle writes

**Validation**

- no accepted lifecycle transition remains with ambiguous owner
- `next`, `verify-work`, and `code-review` relations remain explicit

## Work Package 2: Render And Integration Hook Contract

**Goal**

Specify what render hooks and integration hooks may write, mirror, or never touch.

**Reference implementation bias**

- keep hooks smaller than `planning-with-files`'s aggressive context-reinjection model
- keep integration outputs as evidence, not as workflow verdicts
- use GSD's narrower hook philosophy as the baseline

**Concrete steps**

1. Turn the render-hook permission matrix into an explicit allow/deny table.
2. Turn the integration-evidence matrix into explicit evidence/object-status/forbidden-write tables.
3. Define which metadata fields count as narrow mirror-back and which would illegally become steering state.
4. Define later artifact/report contract touchpoints without reopening `05-02`.

**Expected outputs**

- render-hook permission contract
- integration evidence contract
- narrow metadata mirror-back definition

**Validation**

- render hooks are useful but cannot rewrite control truth
- integration hooks can capture facts without becoming a second control plane

## Work Package 3: Codex Adapter And Runtime Helper Contract

**Goal**

Pin down the boundary between public skill prompts, runtime helpers/scripts, hooks, and Codex adapters.

**Reference implementation bias**

- follow GSD installer patterns for runtime translation
- keep Codex adapter behavior compatibility-oriented
- keep canonical project semantics outside adapters

**Concrete steps**

1. Convert the adapter boundary matrix into an implementation-facing ownership table.
2. Define what belongs in:
   - public command/workflow docs
   - runtime helper scripts
   - hook logic
   - Codex adapter headers/helpers
3. Define explicit anti-patterns:
   - adapter-owned phase routing
   - adapter-owned migration semantics
   - hook-owned verdicts
4. Define the first rewrite wave's ownership-safe file targets.

**Expected outputs**

- adapter/runtime-helper ownership contract
- anti-pattern table
- first rewrite wave handoff notes

**Validation**

- runtime translation and project semantics are not mixed
- future implementation can follow GSD's existing adapter pattern instead of inventing a parallel one

## Integration Pass

Before Phase 6 is considered planning-complete:

1. Phase 5 migration rules still align with Phase 6 ownership rules.
2. Render and integration hooks do not violate the command-owned lifecycle rule.
3. Adapter ownership does not overlap with lifecycle or migration semantics.
4. The resulting handoff is specific enough to begin rewrite without reopening hook ownership at implementation time.

## Deferred To Later Implementation Detail

- exact hook registration format per runtime
- exact prompt wording for hook warnings
- exact aggregation heuristics for code-review subagents
- exact logging and tracing format for hook runs

## Deliverables

- this `06-01-PLAN.md`
- later `06-02` and `06-03` plans for artifact/integration boundaries and Codex compatibility details
- a stable ownership contract for hooks, helpers, and adapters
