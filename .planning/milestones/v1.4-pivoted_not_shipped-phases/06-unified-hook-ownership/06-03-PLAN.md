# Phase 6 Plan 06-03: Codex Compatibility Contract

**Date:** 2026-04-09
**Status:** Draft execution-grade plan

## Objective

Turn the accepted Codex adapter and runtime-helper boundaries into a concrete compatibility contract so `ljx-GSD-*` can be rewritten for Codex without mixing runtime translation, project semantics, hook ownership, or state mutation rules.

## Locked Assumptions

- `.planning/` remains the authoritative root.
- `06-01` remains the source of truth for lifecycle ownership.
- `06-02` remains the source of truth for render/integration hook boundaries.
- `05-02` state-family namespaces and report contracts remain locked.
- Mature GSD installer and Codex adapter behavior should be preferred wherever it already solves the problem cleanly.
- `ljx-GSD` should not invent a second compatibility layer that competes with public workflow specs.

## Scope Boundary

This plan is responsible for:

- defining what belongs in public workflow specs versus runtime helpers versus hooks versus Codex adapters
- defining the first rewrite wave's adapter/header responsibilities
- defining config mutation and hook-registration expectations for Codex
- defining path-resolution, idempotency, and compatibility rules for installer-generated assets
- defining explicit anti-patterns that later implementation must avoid

This plan is not responsible for:

- rewriting the skills yet
- reopening lifecycle or hook ownership semantics
- designing a second runtime beyond the accepted Codex target
- changing the accepted public command surface

## Reference Implementation Bias

Prefer GSD's mature Codex implementation patterns where possible:

1. `getCodexSkillAdapterHeader()` style runtime translation remains the reference for invocation and tool-semantic mapping
2. installer-managed config mutation should remain idempotent and explicit
3. hook registration should remain narrow, deliberate, and path-safe
4. local install path handling should remain cwd-independent
5. command/workflow semantics should remain outside adapters and hooks

Concrete signals from existing GSD implementation:

- `bin/install.js` already separates:
  - markdown/workflow conversion
  - adapter-header injection
  - agent config generation
  - config mutation and hook registration
- GSD's current Codex install keeps a light visible hook surface, mainly `SessionStart`
- GSD already tracks `codex_hooks` ownership and performs idempotent config rewriting
- GSD already protects local hook paths against cwd-sensitive breakage by anchoring commands correctly

## Execution Order

1. **WP1: Public spec vs runtime helper vs hook vs adapter ownership**
2. **WP2: Codex adapter translation and installer/config contract**
3. **WP3: First rewrite-wave compatibility handoff and anti-pattern table**
4. **Integration pass: confirm the full Phase 6 ownership package is implementation-ready**

Reasoning:

- WP1 fixes the conceptual split.
- WP2 fixes the concrete Codex-facing runtime/installer contract.
- WP3 turns those boundaries into a safe handoff for actual rewrite work.
- The integration pass decides whether Phase 6 can be treated as an accepted baseline.

## Work Package 1: Public Spec vs Runtime Helper vs Hook vs Adapter Ownership

**Goal**

Convert the accepted ownership split into an implementation-facing table that later rewrites can follow without ambiguity.

**Concrete steps**

1. Freeze the four-layer split:
   - public command/workflow specs
   - runtime helpers/scripts
   - hooks
   - Codex adapters
2. Define the canonical ownership of:
   - user-facing workflow meaning
   - phase routing and `next`
   - deterministic state-file writes
   - schema normalization and report generation
   - observe/warn/block behavior
   - render behavior
   - invocation syntax translation
   - tool-semantic translation
   - capability detection and fallback UX
3. Define explicit no-go zones:
   - adapters may not own project semantics
   - hooks may not own lifecycle truth
   - runtime helpers may not become hidden planners
   - public workflow specs may not rely on host-specific semantics to stay coherent

**Expected outputs**

- ownership table suitable for rewrite work
- forbidden-overlap table
- one-line implementation rules for each layer

**Validation**

- every responsibility has one clear owner
- no accepted responsibility is split across adapter and workflow layers

## Work Package 2: Codex Adapter Translation And Installer/Config Contract

**Goal**

Define exactly what the Codex adapter layer and installer/config layer should do in the first rewrite wave.

**Concrete steps**

1. Preserve the GSD-style adapter-header responsibilities:
   - invocation parsing
   - `AskUserQuestion -> request_user_input`
   - `Task -> spawn_agent`
   - text-mode / execute-mode fallback
2. Preserve the installer-style config responsibilities:
   - idempotent config mutation
   - explicit `codex_hooks` handling
   - explicit agent-config registration
   - explicit hook registration with ownership markers
3. Preserve the path-safety contract:
   - use absolute paths where Codex requires them
   - ensure local hook commands are cwd-independent
   - keep generated references stable across global/local installs
4. Define the minimal Codex compatibility surface:
   - no hidden adapter-owned state logic
   - no host-specific lifecycle meaning
   - no implicit config mutation outside installer/runtime-helper flows

**Expected outputs**

- adapter responsibility contract
- installer/config mutation contract
- path-resolution and idempotency contract

**Validation**

- the resulting design still looks like a refined version of GSD's mature Codex support, not a parallel compatibility framework
- installer and adapter responsibilities remain compatibility-oriented rather than project-semantic

## Work Package 3: First Rewrite-Wave Compatibility Handoff And Anti-Pattern Table

**Goal**

Make the Phase 6 result concrete enough that actual `ljx-GSD-*` rewrite work can start without reopening runtime-boundary questions.

**Concrete steps**

1. Define the first rewrite wave's safe targets:
   - public command markdown/specs
   - runtime helper scripts for deterministic state/report work
   - narrow hooks for observe/guard/render/evidence roles only
   - installer/adapter helpers for Codex compatibility only
2. Define the explicit anti-pattern table:
   - adapter-owned phase routing
   - adapter-owned migration semantics
   - hook-owned verdicts
   - render hook writing steering state
   - integration hook escalating facts into lifecycle truth
   - ambient hook overuse where command-invoked helpers are cleaner
3. Define the implementation handoff checklist:
   - command/workflow semantics stay in specs
   - state writes stay deterministic and schema-bound
   - hooks stay narrow and non-steering
   - adapters stay translation-only
   - config mutation stays explicit and idempotent

**Expected outputs**

- rewrite handoff checklist
- anti-pattern table
- safe first-wave compatibility target list

**Validation**

- a future implementer can begin rewriting `ljx-GSD-*` skills without needing to reinterpret ownership
- Phase 6 no longer has an unresolved ambiguity that blocks implementation start
