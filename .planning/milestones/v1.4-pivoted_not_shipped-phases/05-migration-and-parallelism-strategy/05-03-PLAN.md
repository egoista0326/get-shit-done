## Work Package 2: Define The `ljx-GSD` Code-Review Quality Gate

**Objective**

Add a phase-aware code-review command that checks implementation logic and implementation correctness for code-bearing phases without collapsing into research review or generic verification.

**Recommended public shape**

| Item | Recommendation |
|---|---|
| public command | `ljx-GSD-code-review` |
| public surface | project/phase quality command; visible in help, not hidden maintenance |
| default invocation | `ljx-GSD-code-review` or `ljx-GSD-code-review 4` |
| optional flags | `--depth=quick|standard|deep`, `--files=...` |
| primary output | phase-local `{phase_num}-CODE_REVIEW.md` |
| structured summary location | mirror only into `.planning/state/phase-records/{phase_id}.json` |
| default route position | after `execute-phase`, before `verify-work`, whenever the phase produced code or changed source files |
| default depth | `standard` |
| default multi-agent policy | `always` |

**Why a distinct command is needed**

`verify-work` is the broad evidence gate. It asks whether the phase is good enough to continue.

`code-review` asks a narrower and different question:

- is the implementation logic sound?
- are there correctness bugs, edge-case failures, insecure patterns, or serious quality issues in the changed code?

Those are not the same judgment:

- a phase can pass user-visible smoke checks and still contain serious implementation errors
- a research phase can produce valid early evidence while still containing fragile code that should not be trusted

Therefore the recommended lifecycle for code-bearing phases is:

```text
execute-phase -> code-review -> verify-work -> next
```

not:

```text
execute-phase -> verify-work only
```

**Distinction from other review commands**

| Command | What It Reviews | Why It Stays Separate |
|---|---|---|
| `ljx-GSD-code-review` | changed source code and implementation logic | correctness and implementation quality gate |
| `ljx-GSD-verify-work` | phase outputs as evidence/acceptance | broader continuation decision |
| `ljx-GSD-research-review` | idea/proposal/result/paper reasoning | intellectual critique, not code audit |
| `ljx-GSD-review-loop` | multi-round adversarial research review workflow | iterative research critique and revision |
| `ljx-GSD-external-review` | external AI review of plans or artifacts | cross-agent perspective, not default implementation audit |

**Phases that should trigger or recommend code review**

| `phase_type` | Default Code-Review Policy |
|---|---|
| `engineering` | required after `execute-phase` |
| `experiment` | required after `execute-phase` |
| `paper` | required when execute output materially changed code, tooling, generation scripts, or compile glue |
| `analysis` | required when execute output materially changed code |
| `refine` | required when execute output materially changed code or prototype implementation |
| `discovery` | required when execute output materially changed code or pilot tooling |

**Recommended implementation model**

Reuse and adapt existing skill logic rather than inventing a second review engine:

1. **Base scoping and workflow pattern from `gsd-code-review`**
   - explicit `--files` override
   - phase-summary extraction
   - git diff fallback
   - per-phase artifact output
2. **Reviewer role pattern from `gsd-code-reviewer`**
   - bug, security, and code-quality scanning
   - depth-aware review modes
   - structured findings
3. **Checkpoint discipline from `requesting-code-review`**
   - review after major implementation batches
   - review before merge/phase completion
4. **Finding-handling discipline from `receiving-code-review`**
   - verify findings, do not blindly accept them
   - fix one item at a time
   - push back when a suggestion is technically wrong
5. **Keep `gsd-code-review-fix` semantics as a later hidden repair workflow**
   - useful, but should not be required for the initial public surface

**Accepted scope and depth semantics**

- `standard` always runs:
  - correctness-and-bugs
  - security-and-robustness
  - simplicity-and-reuse
- `standard` treats efficiency/performance as scope-aware
- `deep` enables efficiency fully and adds wider project-level analysis for:
  - redundancy
  - duplicated logic
  - architectural mismatch
  - whole-project error detection that needs broader repository context
- the primary target is implementation code
- related tests, scripts, and configs are reviewed as supporting evidence
- reviewer lanes should run in spawned subagents with fresh context windows
- one orchestrating agent should merge and deduplicate all findings
- freshness should use `latest-material-scope`
- only the latest review can be fresh
- newer code-review runs make older ones historical
- material code or supporting-file changes inside the reviewed scope invalidate freshness

**Recommended artifact model**

- Human-readable artifact:
  - `.planning/phases/{phase_id}/{phase_num}-CODE_REVIEW.md`
- Structured mirror in phase record only:
  - `code_review.status`
  - `code_review.depth`
  - `code_review.findings_summary`
  - `code_review.last_run_at`
  - `code_review.required_before_verify`

This deliberately avoids adding a new top-level `.planning/state/code-reviews/` family and keeps `05-02` intact.

**Recommended routing rules**

1. If `execute-phase` finishes a code-bearing phase, recommend `ljx-GSD-code-review` before `verify-work`.
2. If `verify-work` sees `code_review.required_before_verify=true` and no fresh code review exists, it should:
   - not silently pass
   - explain the missing quality gate
   - recommend or route to `ljx-GSD-code-review`
3. `ljx-GSD-next` should not skip required code review for code-bearing phases.
4. If unresolved blocking findings remain, both `verify-work` and `next` should refuse continuation and point back to review/fix.
5. If only non-blocking findings remain, `verify-work` may continue with warnings and `next` may remain available.

**Locked defaults**

- `workflow.code_review = true`
- `workflow.code_review_depth = standard`
- `workflow.code_review_multi_agent = always`
- `workflow.code_review_dimensions.correctness = true`
- `workflow.code_review_dimensions.security = true`
- `workflow.code_review_dimensions.simplicity = true`
- `workflow.code_review_dimensions.efficiency = adaptive`
- `workflow.code_review_rerun_after_fix = recommend`
- `workflow.code_review_freshness_policy = latest-material-scope`
- `workflow.code_review_block_policy = { class-based blocking + severity fallback }`
- `workflow.code_review_requirements_by_phase_type = { typed requirement map }`
- `workflow.code_review_deep_policy = { whole-project expansion policy }`

**Locked finding taxonomy**

| Class | Meaning | Default Handling |
|---|---|---|
| `correctness_error` | Functional bug, wrong behavior, broken logic, or missing critical edge-case handling | blocks continuation |
| `security_risk` | Material security or robustness flaw | blocks continuation |
| `obvious_redundancy` | Clear unnecessary duplication inside the new implementation | blocks continuation |
| `wheel_reinvention` | Reimplementation of a repository capability that should have been reused | blocks continuation |
| `material_efficiency_issue` | Meaningful avoidable efficiency loss | severity fallback |
| `maintainability_concern` | Important architecture-fit or complexity concern without immediate breakage | severity fallback |
| `minor_nonimpact_efficiency` | Small inefficiency without meaningful impact | warning only |
| `minor_style` | Low-impact style or convention issue | warning only |

Interpretation:

- reviewer lanes should emit both `class` and `severity`
- class decides blocking before severity does
- severity is only the fallback path for classes like `material_efficiency_issue` and `maintainability_concern`

**Still-open implementation-detail questions**

- aggregation facility details:
  - duplicate finding collapse
  - conflicting reviewer judgment handling
  - phase-summary versus file-level evidence balance
- lane-prompt tuning and aggregation heuristics

**Expected outputs**

- final recommendation that `ljx-GSD-code-review` should exist as a distinct quality-gate command
- integration contract between `execute-phase`, `code-review`, `verify-work`, and `next`
- reuse plan for legacy code-review-related skills
