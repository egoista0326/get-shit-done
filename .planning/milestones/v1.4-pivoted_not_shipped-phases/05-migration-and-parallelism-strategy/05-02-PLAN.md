## Locked Planning Assumptions

1. `.planning/` remains the authoritative root.
2. Root markdown files and phase-local markdown artifacts remain the human-readable operator surface.
3. `.planning/state/` remains the canonical structured machine-state root.
4. `STATE.md` remains a runtime summary mirror, not a replacement for structured state.
5. Backup sessions remain archival evidence only and may not re-enter the default control path after release.
6. `primary` remains rooted at top-level `.planning/` while also having a formal workstream record.
7. Migration report contracts must support both `safe` and `autonomous` without creating two report taxonomies.

**Concrete steps**

1. Freeze the exact directory names above as canonical.
2. Freeze the collection rule:
   - single evolving object families use one file per logical id
   - repeated historical events use a session- or promotion-keyed collection
3. Freeze `primary` handling:
   - the live mainline remains top-level `.planning/...`
   - `.planning/state/workstreams/primary.json` exists as the structured workstream identity only
4. Freeze the migration collection rule:
   - all migration runtime objects live under `.planning/state/migration/`
   - no migration runtime object should be written into phase-local directories
5. Freeze the research subfamily rule:
   - discovery/refine runtime objects live under `.planning/state/research/`
   - domain artifacts still remain phase-local markdown when human-readable output matters

**Expected outputs**

- final namespace table for all migration-relevant structured families
- elimination of singular ad hoc draft names such as `import-session.json` in favor of collection-based families
- stable path contracts that later implementation may treat as non-provisional

## Work Package 2: Lock Human-Readable Report Contracts

**Objective**

Fix the operator-facing report surface so migration, repair, promotion, and adoption flows all expose one stable human-readable contract.

**Locked human-readable report surface**

| Report | Final Path | Purpose | May Control Truth? |
|---|---|---|---|
| migration summary | `.planning/MIGRATION_SUMMARY.md` | Current or most recent migration summary shown before or during import/release | no |
| conflict report | `.planning/CONFLICT_REPORT.md` | Human-readable explanation of blocking and non-blocking migration conflicts | no |
| repair bundle | `.planning/REPAIR_BUNDLE.md` | Visible summary of auto-applicable low-risk repairs and deferred explicit repairs | no |
| suggested branches | `.planning/SUGGESTED_BRANCHES.md` | Human-readable summary of branch candidates that were not promoted into formal workstreams | no |
| promotion diff | `.planning/migration/promotions/{promotion_id}-DIFF.md` | Human-readable comparison between chosen promotion baseline and current primary | no |
| backup conversion report | `.planning/legacy-backups/{source_family}/{session_id}/conversion-report.md` | Archival human-readable record of what one backup/import session did | no |

**Required report contracts**

### `MIGRATION_SUMMARY.md`

Must include:

- import session id
- detected legacy source families
- migration mode and gating mode
- planned normalized outputs
- blocking status summary
- repair-bundle preview
- confirmation boundary and recommended next action

### `CONFLICT_REPORT.md`

Must include:

- import session id
- conflict class and severity
- affected source files or normalized objects
- current winner or unresolved status
- whether the conflict is blocking
- suggested repair path

### `REPAIR_BUNDLE.md`

Must include:

- import session id
- auto-applicable low-risk repairs
- deferred explicit repairs
- `safe` versus `autonomous` execution timing
- post-apply expectations and residual warnings

### `SUGGESTED_BRANCHES.md`

Must include:

- suggested-branch id
- title or semantic theme
- why it was not promoted automatically
- required promotion checks
- reminder that promotion baseline must be user-selected

### Promotion diff report

Must include:

- promotion id
- suggested-branch source id
- chosen baseline mode
- current-primary reference
- phase-chain differences
- active-position differences
- key artifact differences
- expected `next` and workstream-structure impact

### Backup `conversion-report.md`

Must include:

- backup/import session id
- preserved source roots
- normalized outputs produced
- low-risk repairs applied
- unresolved residue and optional gaps
- release verdict or blocked reason

**Concrete steps**

1. Freeze the report filenames above as canonical.
2. Freeze the split between live operator reports and archival backup reports.
3. Freeze the rule that every human-readable report must point back to a structured state object or session id.
4. Freeze the rule that reports may summarize, explain, and recommend, but may not override structured state.

**Expected outputs**

- final migration report-surface contract
- stable operator-facing report names that `help`, `progress`, import, repair, and promotion flows can reference directly

## Work Package 3: Lock Ownership, Write-Order, And Validation Responsibilities

**Objective**

Prevent migration state, reports, backup archives, and phase-local evidence from drifting into overlapping control roles.

**Ownership lock**

| Layer | Owns | Does Not Own |
|---|---|---|
| structured state families under `.planning/state/` | machine truth for migration status, workstream identity, conflict classes, repair bundles, release status, promotion lineage | long-form human narrative, archival originals, phase-local prose evidence |
| root human-readable reports under `.planning/` | operator-facing summaries, explanation, review surface, recommended next actions | authoritative control truth |
| phase-local markdown artifacts | domain evidence and deliverables for discovery/refine/experiment/analysis/paper phases | migration release state, conflict truth, promotion lineage truth |
| backup sessions under `.planning/legacy-backups/` | immutable preserved originals and session-local archival record | active control truth after successful conversion |

**Locked write order**

1. create backup session and preserve originals
2. write normalized structured state
3. write or refresh human-readable migration reports
4. update release-facing `STATE.md` summary
5. only after release criteria pass, unblock normal lifecycle

**Locked validation obligations**

- every human-readable report must reference an existing structured migration object or import session id
- every structured migration object must be derivable from the import session and backup manifest
- no phase-local markdown artifact may be used as the sole migration control truth
- `STATE.md` may summarize migration release state but may not replace `.planning/state/migration/...`
- no default writer may write new control truth back into legacy file locations after cutover

**Concrete steps**

1. Freeze the ownership table above.
2. Freeze the write order above.
3. Freeze the validation checks that later implementation must run before normal lifecycle unlock.
4. Cross-check that all blocked-state and release-stage commands can operate using only the locked families and report contracts.

**Expected outputs**

- final ownership contract
- final write-order contract
- final validation contract for schema/report-layer release
