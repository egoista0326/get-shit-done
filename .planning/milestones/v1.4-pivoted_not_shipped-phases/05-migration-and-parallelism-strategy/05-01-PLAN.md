# Plan 05-01: Define Legacy Import, Validation, And Parallel-Branch Engines

## Goal

Turn the accepted Phase 5 migration semantics into an execution-grade implementation plan for the three engine families that must make `ljx-GSD` cutover safe:

- import and normalization
- validation, conflict handling, and repair
- workstreams, suggested branches, and adoption back into `primary`

This plan is not the final code implementation. It is the execution-grade planning layer that should drive the later rewrite/spec work without reopening the already accepted migration semantics.

## Scope Boundary

This plan is responsible for:

- turning the five Phase 5 research matrices into concrete implementation work packages
- defining work-package order and dependencies
- specifying the concrete outputs each package must produce
- identifying the state families, artifact families, and command surfaces each package touches
- making the main migration risks explicit before implementation begins

This plan is not responsible for:

- final hook ownership
- optional provider packs
- rewriting installed skills in this step
- re-opening already accepted semantic decisions unless a contradiction is discovered

## Locked Planning Assumptions

1. `.planning/` remains the authoritative root.
2. Legacy inputs are readable during migration, but post-conversion writes target only new `ljx-GSD` truth by default.
3. Migration is one-shot and guided by default.
4. Migration blocks normal lifecycle commands until validation passes.
5. `primary` remains at root `.planning` scope after cutover, while still having a formal structured workstream identity.
6. No whole-workstream implicit merge-back into `primary` is allowed.
7. Useful workstream output re-enters the mainline only through explicit adoption/import of bounded artifacts.

## Execution Order

The work packages are not independent. Execute them in this order:

1. **WP1: Import And Normalization Engine**
2. **WP2: Validation, Conflict, And Repair Engine**
3. **WP3: Workstream, Suggested-Branch, And Adoption Engine**
4. **Integration Pass: Cross-engine consistency review**

Reasoning:

- WP1 defines what objects and artifact/state families exist after import.
- WP2 depends on WP1 because validation and repair need a concrete normalized target model.
- WP3 depends on both WP1 and WP2 because imported workstreams, suggested branches, and adoption flows must operate on normalized objects and validated branch semantics.
- The integration pass is required so the three engines do not drift on object names, state ownership, or command-surface behavior.

## Work Package 1: Import And Normalization Engine

**Objective**

Specify exactly how legacy GSD and Auto artifacts are detected, backed up, classified, normalized, and written into new `ljx-GSD`-native `.planning` truth.

**Primary source material**

- Matrix 1: `Legacy Object Mapping Matrix`
- Matrix 3: migration stages and output artifacts
- accepted backup, cutover, and one-shot import decisions

**Responsibilities**

- detect whether a project is legacy-bearing and migration-required
- classify source families: GSD, Auto, mixed
- generate the guided import summary before mutation
- create the legacy backup session
- normalize legacy objects into target artifact/state families
- write new truth in the correct order

**Concrete steps**

1. Lock the set of normalized target families that WP1 is allowed to write:
   - root project docs
   - phase-local artifacts
   - structured phase records
   - structured research families
   - structured migration families
   - structured workstream/session families that are required for cutover
2. Define the legacy detection and preflight routine:
   - which file families count as legacy signals
   - what qualifies as “migration required”
   - how mixed GSD + Auto projects are classified
3. Define the guided import summary shape:
   - detected legacy sources
   - planned backup location
   - planned normalized outputs
   - conflicts already visible before writing
   - low-risk repair bundle candidates
4. Define the backup writer:
   - exact directory layout under `.planning/legacy-backups/`
   - preservation of original relative paths under `original/`
   - manifest responsibilities
   - conversion-report responsibilities
5. Define the normalization pipeline by legacy family:
   - GSD root project artifacts
   - GSD phase artifacts
   - GSD workstream/workspace artifacts
   - Auto idea/refine artifacts
   - Auto experiment/review/claim artifacts
   - Auto paper artifacts
6. Define the write order for safe cutover:
   - create backup session first
   - write normalized state/artifacts second
   - write migration-side reports third
   - only then update release-facing runtime summary
7. Define which legacy inputs become:
   - first-class normalized objects
   - supporting evidence only
   - suggested branches
   - archived residue

**Expected outputs**

- finalized import-engine specification section or document
- finalized import-summary schema
- finalized backup-session schema
- finalized normalization-write-order description
- explicit target-family list used by later engines

**Validation**

- every row in Matrix 1 maps to one of the declared normalized targets
- no legacy family remains ambiguously “half-authoritative”
- backup is always created before state mutation
- the write order preserves `read-old / write-new only`

**Main risks**

- final target state families are still provisional in a few places
- mixed GSD/Auto projects may expose artifact overlaps that look similar but belong to different truth layers

## Work Package 2: Validation, Conflict, And Repair Engine

**Objective**

Specify how the migrated project is checked, how conflicts are classified, how repair suggestions are generated, and what may be auto-applied versus requiring explicit repair.

**Primary source material**

- Matrix 2: `Conflict And Repair Matrix`
- Matrix 3: `Migration Gating Matrix`
- accepted `safe` versus `autonomous` repair policy

**Responsibilities**

- validate normalized state before release
- classify blocking versus non-blocking conflicts
- generate structured repair suggestions
- define low-risk auto-repair boundaries
- define release gating and blocked-state behavior

**Concrete steps**

1. Define the validation pass routine:
   - what must be checked after normalization
   - what constitutes a blocking failure
   - what can remain warning-only
2. Convert Matrix 2 conflict classes into detector families:
   - graph failures
   - control-field conflicts
   - artifact attachment failures
   - structured-state integrity failures
   - backup/report completeness failures
   - residue-only non-blocking issues
3. Define the conflict report structure:
   - conflict class
   - severity
   - affected source files/objects
   - new-state winner or unresolved status
   - suggested repair action
4. Define the repair-suggestion engine:
   - when suggestions are generated
   - how they reference the affected normalized objects
   - how they group into one visible repair bundle under `safe`
5. Define the low-risk auto-repair boundary:
   - index repair
   - reference relinking
   - non-semantic naming normalization
   - duplicate low-value archive marking
   - metadata cleanup
6. Define the explicit-repair boundary:
   - phase graph mutation
   - `phase_type` mutation
   - core artifact reassignment
   - control-truth overwrite
   - backup-completeness changes
   - routing-affecting fixes
7. Define the release gate:
   - when migration blocking lifts
   - which reports must exist
   - what `STATE.md` and `progress` must be able to explain before release

**Expected outputs**

- finalized validation-and-conflict specification section or document
- finalized conflict-report schema
- finalized repair-bundle schema
- finalized release-pass checklist

**Validation**

- all blocking/non-blocking decisions from Matrix 2 remain preserved
- `safe` and `autonomous` differ only where already accepted, not ad hoc
- no repair path silently mutates control truth beyond the accepted low-risk boundary

**Main risks**

- over-broad auto-repair would silently mutate the new truth model
- under-specified release criteria would let half-migrated projects escape into normal lifecycle

## Work Package 3: Workstream, Suggested-Branch, And Adoption Engine

**Objective**

Specify how imported parallel lines become `primary`, formal workstreams, suggested branches, or archived residue, and how later promotion/adoption flows behave.

**Primary source material**

- Matrix 4: `Workstream Migration Matrix`
- Matrix 5: `Promotion Provenance Matrix`
- accepted `primary` visibility and adoption rules

**Responsibilities**

- define the post-cutover `primary` model
- define imported workstream creation
- define suggested-branch storage and promotion
- define primary-safe workstream commands
- define adoption from workstreams back into `primary`

**Concrete steps**

1. Define the special role of `primary`:
   - root `.planning` remains the live mainline
   - structured `primary` record exists for workstream-aware features
   - `primary` appears explicitly in workstream UX
2. Define legal versus illegal `primary` workstream operations:
   - legal: list, status, progress, switch, resume
   - illegal: complete, remove, rename, archive
3. Define imported-workstream creation:
   - threshold tests for `clear + stable + separable`
   - semantic naming
   - initial activation rules
4. Define suggested-branch storage and visibility:
   - migration-side index
   - human-readable summary
   - exclusion from default `next` routing
5. Define promotion mechanics:
   - explicit promotion flow
   - user-chosen baseline
   - required diff summary
   - inactive-by-default promoted workstream
   - required provenance payload
6. Define workstream adoption back into `primary`:
   - no generic whole-workstream merge-back
   - adopt/import bounded outputs only
   - default landing via new future phase
   - inserted phase only under accepted urgent threshold
   - semantic `phase_type` mapping for adopted outputs
   - multi-output splitting and ordering rules
   - multi-phase adoption bundle confirmation

**Expected outputs**

- finalized workstream and suggested-branch specification section or document
- finalized promotion-record schema
- finalized adoption-bundle schema
- finalized `primary` workstream command-surface rules

**Validation**

- `primary` remains visible without becoming a duplicated physical branch
- suggested branches never behave like active workstreams unless explicitly promoted
- adoption flows cannot silently mutate the current active phase
- multi-output adoption preserves clean `phase_type` routing

**Main risks**

- accidentally treating `primary` like an ordinary workstream would weaken mainline safety
- adoption could become an implicit merge-back if the bundle and phase-creation rules are not explicit enough

## Integration Pass

After the three work packages are specified, run one explicit integration pass.

**Integration checks**

1. The same target object names appear consistently across WP1, WP2, and WP3.
2. The same command surfaces are not described with contradictory blocking behavior.
3. `progress`, `next`, `pause-work`, `resume-work`, and `workstreams` can all explain the migrated project coherently.
4. No workstream rule contradicts the migration-blocking or release rules.
5. No adoption rule contradicts the typed-phase routing model or the accepted public interfaces.

**Integration outputs**

- a reconciled migration-engine plan with no naming drift
- a short list of unresolved schema-lock items, if any
- a short list of items explicitly deferred to Phase 6 hook ownership
