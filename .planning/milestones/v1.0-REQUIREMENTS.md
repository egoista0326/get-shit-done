# Requirements: ljx-GSD

**Defined:** 2026-04-09
**Core Value:** One authoritative research workflow system should be able to take a direction from idea to evidence to paper without losing state, duplicating control planes, or confusing which artifact is the truth.

## v1 Requirements

### State Model

- [ ] **STATE-01**: The system defines exactly one authoritative workflow state model for sessions, plans, runs, reviews, and papers.
- [ ] **STATE-02**: The system separates authoritative control state from derived markdown mirrors and summaries.
- [ ] **STATE-03**: The system supports resumable recovery without requiring terminal transcript scraping when structured state exists.
- [ ] **STATE-04**: The system supports artifact identity independent of filesystem path.

### Workflow Surface

- [ ] **WF-01**: The system defines a clear public `ljx-GSD-*` command taxonomy with no duplicate top-level responsibilities.
- [ ] **WF-02**: The system preserves an outer orchestration flow for project/phase style execution.
- [ ] **WF-03**: The system exposes research-native flows for ideation, method refinement, experiment planning, review loops, claim gating, and paper generation.
- [ ] **WF-04**: The system defines which capabilities are first-class commands versus internal adapters/stages.

### Migration

- [ ] **MIG-01**: The system defines a migration mapping from current GSD artifacts and commands to `ljx-GSD-*`.
- [ ] **MIG-02**: The system defines a migration mapping from current Auto research artifacts and workflows to `ljx-GSD-*`.
- [ ] **MIG-03**: The system preserves a staged transition path rather than requiring an all-at-once cutover.
- [ ] **MIG-04**: The system defines how legacy `.planning/` and Auto root-level artifacts remain readable during transition.

### Hooks And Ownership

- [ ] **HOOK-01**: The system defines a single hook ownership model for lifecycle, artifact-render, and integration hooks.
- [ ] **HOOK-02**: The system prevents multiple components from writing the same authoritative field family.
- [ ] **HOOK-03**: The system defines Codex-specific compatibility and write boundaries explicitly.

### Research Execution

- [ ] **RSCH-01**: The system models experiments as campaigns plus runs with structured status and evidence.
- [ ] **RSCH-02**: The system models review loops as first-class state objects rather than loose markdown accumulation.
- [ ] **RSCH-03**: The system can represent the path from idea to claim to paper without switching orchestration systems.
- [ ] **RSCH-04**: The system supports workstreams for parallel research tracks without corrupting shared state.

### Implementation Completion

- [x] **IMPL-01**: The system lands a shared authoritative runtime substrate under `.planning/state` plus common config, phase, and workstream resolution helpers.
- [x] **IMPL-02**: The system completes the typed-phase lifecycle shell for `discuss-phase`, `plan-phase`, `execute-phase`, and lifecycle-aware `next`.
- [x] **IMPL-03**: The system integrates code review and verification as first-class lifecycle quality gates with enforced freshness and blocking semantics.
- [x] **IMPL-04**: Discovery and refinement workflows graduate from thin bridge wrappers to full typed-phase implementations with explicit handoffs.
- [x] **IMPL-05**: Experiment, review, and claim workflows share one coherent execution, evidence, and judgment model.
- [ ] **IMPL-06**: Paper and rebuttal workflows complete without inventing a second submission control plane.
- [x] **IMPL-07**: Workstream mutation and roadmap mutation commands preserve primary/mainline semantics and typed dependency safety.
- [x] **IMPL-08**: Migration cutover can import legacy artifacts into the canonical runtime state and verify parity before release.

## v2 Requirements

### Optional Integrations

- **V2-01**: Provider-specific review backends can be plugged in without changing the core workflow model.
- **V2-02**: Grant, domain-specific, and notification workflows can be attached as optional vertical packs.
- **V2-03**: Publishing, PR, and collaboration features can be reintroduced as optional research-adjacent packs.

## Out of Scope

| Feature | Reason |
|---------|--------|
| Immediate replacement of globally installed production skills | Preview/runtime implementation happens in this repo first; global cutover waits for parity verification |
| Full UI/product workflow restoration from current GSD | Research orchestration is the current target |
| Multiple competing state roots in the final architecture | Violates the main objective of unification even if everything stays under `.planning/` |
| Mandatory provider-specific review engines | Would over-couple the base system too early |
